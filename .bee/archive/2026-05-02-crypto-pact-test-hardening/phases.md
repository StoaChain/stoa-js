# Phases: crypto-pact-test-hardening

## Overview
v2.2.0 is an additive minor release closing three of the four high-additive audit findings (F-CORE-009, F-CORE-011, F-CORE-012). The fourth finding, F-CORE-010 (Pact-code injection escaping), was reviewed during spec discovery and explicitly rejected as consumer-side overreach: chain-side Pact compilation already validates the formats the proposed validator would mirror, and the proposed charset regex did not even match existing Smart and Standard account prefixes. The remaining two phases are split along a clean implementation boundary: Phase 1 is production code in the crypto subpath only; Phase 2 is test coverage plus the release artifacts (version bump, changelog, README). Per the atomic-ship contract, both phases ship in a single commit and a single annotated tag — Phase 1's new error classes are exercised by Phase 2's new test cases in the same release, and the workflow gate enforces version parity across `package.json`, the changelog leading heading, and the two README anchors before the publish step runs.

## Phase 1: Crypto error taxonomy and smartDecrypt timing-leak fix
**Description:** Introduce a typed error taxonomy across the crypto subpath and remove the timing side channel in the smart-decrypt dispatcher. Three new public error classes are added to the crypto barrel: a wrong-password class, a corrupt-envelope class, and an unsupported-format class. The v1 and v2 decrypt paths are updated to classify their failures by which decode step failed (JSON-parse boundary versus AES-GCM authentication-tag failure) and to throw the corresponding class. The smart-decrypt dispatcher is rewritten to use the deterministic shape predicate as its single branch, eliminating the v1-then-v2 fallback chain and the wall-clock differential it produced on a wrong-password input. The internal v1-fallback path inside the direct v2 decrypt entry point is preserved unchanged. Diagnostic console output in the v1 catch-alls is removed; the structured cause attached to the thrown error replaces it. No tests are added in this phase — Phase 2 owns the assertions for the new error classes per the atomic-ship contract.
**Deliverables:**
- Three new public error classes added to the crypto barrel as additive exports observable in the published distribution.
- Decryption error classification across the v1 and v2 paths so that JSON-parse failures, authentication-tag failures, and other failures each map to the appropriate class.
- A rewritten smart-decrypt dispatcher that uses the existing shape predicate as its single branch and routes to exactly one decryption path with no fallback.
- Removal of the diagnostic console output from the v1 catch-alls; structured causes carry the failure detail forward.
- Preservation of the internal v1-fallback branch inside the direct v2 decrypt entry point so that consumers who call that entry point directly with a v1 blob keep the existing belt-and-suspenders behavior.
**Dependencies:** None (first phase).

## Phase 2: Test coverage for critical untested surfaces and release artifacts
**Description:** Add deterministic test coverage for the four critical untested surfaces identified by audit findings F-CORE-011 and F-CORE-012, extend five existing test files with the additional cases required by the new Phase 1 surface and by the audit's coverage targets, and produce all release artifacts (version bump, changelog entry, README updates) in the same commit so that the workflow gate's three-check version-parity criterion is satisfied. Four new test files are introduced: pact-reader seam contract, wallet-builder vendor-vector coverage, transaction-errors branch coverage, and codex seed-type migration round-trip. Five existing test files are extended: encryption (tampered v2 envelope, smart-encrypt version selector, wrong-password class assertion), encryption-upgrade (corrupted-v1 blob through smart-decrypt does not fall through to v2 KDF), codex-codec (forward-compat unknown-field preservation), cfm-builders (defensive amount validation on two representative builders), and pact-format (leading-zeros, EU-separator, trailing-zeros edge cases). The test count grows by at least 29 cases over the v2.1.2 baseline. Total test wall-time stays under 30 seconds on Node 22 / Ubuntu. The release artifacts include the version bump from 2.1.2 to 2.2.0, the new leading changelog entry citing the three closed findings and the F-CORE-010 rejection rationale, and the README updates required by the locked maintenance rule (Status block, version history paragraph, test-count refresh, submodule table row, optional "what's new" block). A verification gate runs typecheck, the full test suite, and the build before the phase is considered complete; the pre-existing locale-sensitive gas-test failure on Windows non-en-US continues to be the sole allowed non-zero exit, matching the criterion locked in v2.1.2.
**Deliverables:**
- A new pact-reader test surface that exercises the default reader, the setter swap, the argument-forwarding read function, the replace-not-stack behavior, and the getter — all using stub readers, no real network calls.
- A new wallet-builder test surface that pins vendor public-key vectors across the three supported seed types, asserts mnemonic-length and seed-type mismatch rejection, asserts the validity check across valid, invalid, and wrong-length inputs, and asserts the length-bounded mnemonic generation.
- A new transaction-errors test surface that covers every documented branch of the signing-error helper and the simulation-error helper, plus an independent gas-extraction regex assertion and a user-facing formatter shape assertion.
- A new codex seed-type migration test surface scoped to integration round-trip plus the unknown-input invariant, deliberately non-overlapping with the existing migration unit tests in the codex-codec test file.
- Extensions to the encryption test file asserting the tampered-v2 path, the smart-encrypt version selector, and the wrong-password class introduced in Phase 1.
- Extension to the encryption-upgrade test file asserting that a corrupted-v1 blob passed to smart-decrypt under the new single-path dispatch does not silently fall through to a v2 key derivation.
- Extension to the codex-codec test file asserting forward-compat unknown-field preservation through the deserialization round-trip.
- Extensions to the cfm-builders test file asserting defensive amount validation on two representative builders.
- Extensions to the pact-format test file asserting the documented behavior under leading-zeros, EU-separator, trailing-zeros, and scientific-notation inputs.
- Package version bumped to 2.2.0.
- Changelog leading entry describing the additive nature of the release, listing the three new error classes under "added (public surface)", citing the three closed audit findings under "fixed", including a "rejected (decisions log)" section recording the F-CORE-010 review-and-reject decision with the chain-side-validation rationale, and concluding with a stats block citing the new test count and the no-public-API-removal note.
- README updated per the locked maintenance rule: Status block leads with 2.2.0, version history extended with a v2.2.0 paragraph, test-count references refreshed, submodule table row for the crypto subpath cites the three new error classes, optional "what's new" usage example included.
- Workflow-gate compliance: the three checks (Status block, version history paragraph, changelog first heading) all reference 2.2.0 so that the publish workflow does not block the annotated tag.
- A verification gate that runs typecheck, the full test suite, and the build, accepting the pre-existing locale-sensitive gas-test failure on Windows non-en-US as the sole allowed non-zero exit.
**Dependencies:** Phase 1. The encryption test extensions in this phase assert the new error classes introduced in Phase 1, so Phase 2 cannot land independently of Phase 1.

## F-CORE-010 Rejection Note
F-CORE-010 (Pact-code injection escaping via a `pactString(s, fieldName)` validator with a Unicode charset regex) was originally Phase 2 of the underlying audit bundle. It was reviewed during spec discovery and rejected. Reasons: (a) chain-side Pact compilation and simulation already validate account format, token-ID format, and DALOS charset at submission time, so consumer-side mirroring duplicates logic; (b) the proposed regex did not include the Σ. and Ѻ. account prefixes already in production use, which means landing it would have broken every existing Smart and Standard account name; (c) DALOS may add charset characters in the future, and a hardcoded mirror would drift from chain reality; (d) a narrower blocklist variant was also rejected on the same theoretical-only-defense argument. The audit-spec source remains under `bundles/high-additive/` with a "rejected" note for traceability. Future audit-specs may revisit only if a non-chain-validated context emerges (for example, a HUB-side automation path that bypasses Pact compilation). This is also recorded in the changelog "rejected (decisions log)" section for the v2.2.0 release.
