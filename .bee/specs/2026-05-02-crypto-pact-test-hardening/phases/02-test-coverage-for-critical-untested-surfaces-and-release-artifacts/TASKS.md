**Locked decisions inherited from Phase 1 TASKS.md (CRITICAL — read before authoring REQ-09 / REQ-10 tests):**
- **AES-GCM ambiguity:** `WrongPasswordError` is thrown for ANY AES-GCM auth-tag failure (`error?.name === "OperationError"`), regardless of whether the underlying cause is a wrong password OR ciphertext tampering with the correct password. AES-GCM produces the SAME signal for both cases by spec (W3C WebCryptoAPI). `CorruptEnvelopeError` is reserved for envelope-PARSING failures (JSON.parse, atob outer-decode, envelope-field-shape mismatch including non-object parsed values), NOT ciphertext tampering. Phase 2 REQ-09 may NOT add a test asserting `CorruptEnvelopeError` for ciphertext tampering — such a test would lock an undeliverable contract.
- **REQ-09(1) wording (locked iter 2):** the corrupt-envelope test uses a **structurally-malformed envelope missing the `ciphertext` field** (e.g., `btoa(JSON.stringify({v:2, iv:"...", salt:"..."}))`). The original v-flip wording is undeliverable: flipping `v` from 2 to anything else routes to V1 (which structurally subsumes V2 minus the `v` field), V1 decodes the fields successfully, runs PBKDF2-SHA256/10k on V2-encrypted ciphertext (wrong KDF), AES-GCM auth-tag fails → `OperationError` → `WrongPasswordError`, NOT `CorruptEnvelopeError`. The structurally-malformed-envelope test makes the inner `b642ab(parsed.ciphertext)` call throw on `atob(undefined)` → reliably reaches `CorruptEnvelopeError` via Phase 1's locked envelope-shape mismatch routing.
- **The 3 new error classes** (`WrongPasswordError`, `CorruptEnvelopeError`, `UnsupportedFormatError`) come from Phase 1 — Phase 1 created them in `src/crypto/errors.ts` and re-exported from `src/crypto/index.ts`. **Import path for Phase 2 in-repo tests (LOCKED iter 2 per F-008): use the relative form `from "../src/crypto"` ONLY.** The package-name self-reference `@stoachain/ouronet-core/crypto` is REJECTED for in-repo test files because the package's `exports` map (`package.json:8-32`) routes `./crypto` to `dist/crypto/index.js` which does NOT exist before `npm run build`, and `npm test` runs BEFORE `npm run build` in the verification gate (T2.13). The `@stoachain/ouronet-core/crypto` form IS used in published documentation (e.g., the README "What's new" code example) where end-consumers import from the published distribution — but that is documentation, not test code.
