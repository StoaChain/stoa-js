# Phases: medium-and-low-audit-closures

## Overview
v2.3.0 is an additive minor release closing all 13 audit findings in the MEDIUM and LOW tiers from the 2026-04-30 audit cycle. The work splits across seven phases ordered by risk, not by dependency: a mechanical refactor first (Phase 1) to establish pipeline confidence on a zero-behavior-change baseline; the localized codec and resolver contract tightening (Phase 2); the broader guard-classification semantics work (Phase 3); the documentation refresh that carries no code risk (Phase 4); the small interaction-file catch-block cleanup (Phase 5); the central logger seam plus the tier-semantics JSDoc (Phase 6) which touches the package exports map and introduces a new subpath; and the release artifacts plus verification gate (Phase 7). The atomic-ship contract holds: all seven phases land in a single v2.3.0 commit and a single annotated tag, satisfying the publish-workflow's three-check version-parity gate at the changelog leading heading and the two README anchors. Cross-phase note (LOCKED iter 2 per spec-review I-001): Phase 5's catches route via the central logger seam's error method (REQ-10 LOCKS this regardless of phase ordering — both phases land in the same atomic commit, so Phase 5 has a HARD source-level dependency on Phase 6's `src/observability/{index.ts,logger.ts}` files existing).

## Phase 1: `safeCreationTime` DRY refactor (F-CORE-015)
**Description:** Pure mechanical deduplication of the eleven private `function safeCreationTime` declarations across the interactions module. The canonical declaration in the pact-format module is the single source of truth. Each of the eleven interaction files removes its private copy and gains an import for the canonical symbol. Because the eleven inline copies are byte-identical to the canonical body, this phase produces zero behavior change observable to consumers and zero test count change.
**Deliverables:**
- Removal of the eleven private `function safeCreationTime` declarations across the interactions module (eleven files affected — activate, addLiquidity, coil, crossChain, dex, guard, kpay, ouro, pension, urStoa, wrap function files).
- Import of the canonical symbol from the pact subpath in each of the eleven affected files (added to an existing pact-subpath import line where one is already present, or as a new import where the file did not previously pull from the pact subpath).
- Verification that exactly one `function safeCreationTime` declaration remains in the source tree after the phase (the canonical body in the pact-format module — the eleven removed bodies were byte-identical to it, which is what makes this refactor safe).
- Zero behavior change. Zero test count change. Zero new test cases added by this phase.
**Dependencies:** None (first phase, lowest-risk).

## Phase 2: Codex shape validation and foreign-key resolver contract (F-CORE-013, F-CORE-014)
**Description:** Tighten two domain contracts at the boundary of the codex backup format and at the entry point of the codex signing strategy. The codex deserializer gains runtime shape checks for its known top-level array and object fields, with name-only error messages that do not echo field values. The codex signing strategy gains a pre-flight check that fails fast when a foreign-key transaction reaches a resolver that does not implement the foreign-key request method. The interface JSDoc for the foreign-key request method is clarified to state the runtime contract.
**Deliverables:**
- Runtime shape validation in the codex deserializer for the three array fields (kadena wallets, ouronet wallets, address book) and the one object field (UI settings), positioned after the existing version check and before the typed cast.
- Domain-prefixed error messages on validation failure that name the offending field but never echo its value, preserving the secrets-stay-out-of-telemetry boundary.
- Forward-compatibility preservation: extra unknown top-level fields continue to survive the deserialize round-trip exactly as they do today (the v1.2 envelope contract is unchanged).
- A pre-flight check in the codex signing strategy at the entry point of its execute path: when a transaction requires a foreign-key signer and the configured resolver does not implement the foreign-key request method, throw a precise error before the universal sign helper is invoked.
- Clarified JSDoc on the foreign-key request method of the resolver interface stating the optional-at-the-interface-but-required-at-execute-time-when-needed contract.
- At least three new test cases covering the new contract surface (shape validation hit on each of the three array fields, name-only-no-value-echo assertion on the validation error, foreign-key pre-flight error on a resolver that omits the request method).
**Dependencies:** None on Phase 1 (independent areas of the codebase). May land before, after, or in parallel with Phase 1 from a code-conflict standpoint, but the risk-ordered sequencing places it second.

## Phase 3: Guard hardening and typed predicate error (F-CORE-016a/b/c, F-CORE-017)
**Description:** Tighten guard classification, normalize the keyset-reference field at the chain-IO boundary, document the four reachable states of the Smart Ouronet Account auth-paths analysis, and replace the silent console diagnostic on unrecognized predicates with a typed error class folded into a structured analysis bit. This phase introduces the FIRST of two NEW PUBLIC SURFACES in the release: a typed error class re-exported from the guard subpath.
**Deliverables:**
- Tightened guard-kind classification requiring the FULL minimal shape per kind (capability needs name, arguments, and pact identifier; user needs function and arguments; keyset needs predicate and keys; keyset-reference accepts either casing of the reference field). Under-specified payloads classify as `unknown` instead of silently mis-classifying.
- A keyset-reference normalization helper applied at the resolve-guard boundary so internal code only sees the camelCase form. The helper maps the lowercase chain-native variant at the chain-IO boundary.
- JSDoc enumeration of the four reachable states of the Smart Ouronet Account auth-paths analysis result. An optional `firstSignableButUnsatisfied` field added to the analysis surface naming the index of the first signable-but-unsatisfied path.
- A NEW typed error class (additive public export from the guard subpath) thrown by the threshold-computation helper when it encounters an unrecognized predicate. The general-purpose analyze function catches the class and folds it into a structured `predicateRecognized: false` bit on the returned analysis. The previous silent console diagnostic is removed.
- At least three new test cases covering the new contract surface (under-specified shape classifies as `unknown`, lowercase keyset-reference normalizes to camelCase at the boundary, unrecognized predicate produces both the typed error from the threshold helper and the structured bit on the analyze result).
**Dependencies:** None on Phases 1 or 2. Independent area of the codebase.

## Phase 6: Tier semantics JSDoc and central logger seam (F-CORE-020, F-CORE-022)
**Description:** Document the canonical tier mapping on the pact-reader and the raw calibrated reader, and introduce the central logger seam — the SECOND of two NEW PUBLIC SURFACES in the release. The logger seam is exposed under a NEW `./observability` subpath in the package exports map and mirrors the existing pact-reader injection pattern. Every direct console-warn and console-error call site in the source tree is rerouted through the new seam.
**Deliverables:**
- JSDoc on the pact-reader and the raw calibrated reader enumerating the canonical tier mapping (T1=balance, T2=preview, T3=metadata, T7=very-static, and so on, matching OuronetUI's reader) and clarifying that the default reader accepts and ignores the tier argument. Cross-reference to the pact-reader injection seam for consumers wanting cache-aware behavior. The optional development-mode warning on tier-less reads is DROPPED from scope (LOCKED iter 1 per spec-review advisory #2) — JSDoc-only clarification is sufficient; runtime warnings are out of scope.
- A NEW central logger seam exposed under a NEW `./observability` subpath in the package's exports map. The seam exports a `Logger` type carrying a `warn` method and an `error` method (each accepting a message string followed by additional unknown arguments), a `setLogger` mutator that swaps the active logger, and a `getLogger` accessor that returns the currently-configured logger. Module-private state mirrors the pact-reader injection-seam pattern.
- Default routing: the default logger forwards `warn` to `console.warn` and `error` to `console.error`. Consumers who do nothing observe behavior identical to the previous direct calls.
- Null-rejection contract: the setter throws `TypeError` with message exactly `setLogger requires a non-null Logger` on null or undefined input — the seam refuses to install a non-functional logger silently (see REQ-13 for exact class and message text; LOCKED iter 1 per spec-review I-003). This is part of the runtime contract and is asserted in the tests.
- New `./observability` subpath added to the package's exports map (entry pointing at the compiled JavaScript artifact and types pointing at the compiled declaration artifact, mirroring the existing subpath export shape).
- Sweep of every direct `console.warn` and `console.error` call in the source tree (approximately twenty-five sites across roughly eight files per the audit) replaced with calls routed through the logger accessor. Verifiable post-phase by a search for direct console-warn or console-error calls in the source tree returning zero matches.
- At least three new test cases covering the new contract surface (default getter returns the console-routing default, setter swaps the active logger and getter returns the new one, setter rejects null with the documented error class).
**Dependencies:** None on Phases 1-5 strictly for Phase 6's own deliverables, but the package-exports-map edit and the new subpath want a clean prior state, which is why this phase is sequenced after the other code-touching phases. **Phase 5 has a HARD source-level dependency on Phase 6's `src/observability/{index.ts,logger.ts}` files** (LOCKED iter 2 per spec-review I-001) — Phase 5's catches route via `getLogger().error(...)` from `../observability` per the I-004 lock; the conductor handles intra-commit wave ordering so Phase 6's source files exist on disk before Phase 5's tests run.
