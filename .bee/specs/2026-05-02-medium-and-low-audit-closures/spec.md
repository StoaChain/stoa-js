# Specification: medium-and-low-audit-closures

## Goal
Ship `@stoachain/ouronet-core` v2.3.0 — an additive minor release that closes the entire MEDIUM tier (M1, 7 findings) and the entire LOW tier (M2, 6 findings) from the 2026-04-30 audit cycle in a single atomic commit + tag. The release tightens domain contracts (codex shape validation, foreign-key resolver pre-flight, guard classification, predicate recognition), removes one mechanical duplication (`safeCreationTime` in eleven interaction files), refreshes documentation, normalizes catch-block conventions, and introduces a new central logger seam under a new `./observability` subpath. M3 (fabricated-fallbacks, BREAKING) is deferred to a separate v3.0.0 spec.

## User Stories
- As a downstream consumer importing a codex backup, I want malformed envelopes to be rejected by name-only error messages (no value echo) so that the integrity boundary is honest about what's wrong without leaking secrets into telemetry.
- As an integrator building a server-side resolver that does not handle foreign keys, I want a precise pre-flight error the moment a foreign-key transaction reaches the signing strategy so that I see "this resolver lacks `requestForeignKey`" instead of an opaque deep-stack failure.
- As a Smart Ouronet Account auth consumer, I want under-specified guard payloads to classify as `unknown` rather than be silently mis-classified so that my auth-path resolver surfaces the ambiguity to the user instead of picking a wrong branch.
- As a guard-analysis consumer, I want unrecognized predicates to be flagged via a typed error class and a structured `predicateRecognized: false` bit on the analysis result so that I can branch on the condition cleanly instead of reading console output.
- As a maintainer of the OuroNet ecosystem, I want a single central logger seam — mirroring the existing `setPactReader` injection pattern — so that every `console.warn` and `console.error` site in the library can be redirected to a consumer-supplied sink (Sentry, structured logger, suppression) without forking the library.
- As a downstream consumer reading the README and CONTEXT documents, I want both refreshed to current v2.2.0 reality so that the "what does this library do today" answer matches what's actually shipping.
- As a contributor reading the interactions code, I want catch-block conventions to be consistent within each file so that I'm not asking "why does line 1842 swallow but line 1916 logs" on every code review.
- As a release engineer, I want all 13 finding closures to land in a single v2.3.0 commit + tag, with the publish-workflow's three-check version-parity gate satisfied byte-identically, so that the atomic-ship contract holds and the publish step does not block on a stale README anchor or changelog heading.

## Specific Requirements

### Mechanical refactor — `safeCreationTime` deduplication (M1 Phase 1)
- Eleven interaction files currently each carry a private `function safeCreationTime` declaration whose body is byte-identical to the canonical declaration in the pact-format module. The canonical declaration is the single source of truth and remains unchanged.
- After this phase, only the canonical declaration exists. Every interaction file that previously declared a private copy now imports the canonical name from the pact subpath. Any file that did not already pull from the pact subpath gains a new import for the symbol; files that already imported from it gain the symbol on the existing import line.
- Behavior is byte-identical because the imported body is the same as the removed inline copies. The phase produces zero test count change and zero behavior change observable to consumers.
- Verifiable post-phase by counting `function safeCreationTime` declarations across the source tree and confirming exactly one remains (the canonical).

### Codex shape validation and foreign-key resolver contract (M1 Phase 2)
- The codex deserializer gains runtime shape checks immediately after the existing version check. The `kadenaWallets`, `ouronetWallets`, and `addressBook` fields must be arrays; the `uiSettings` field must be an object. On invalid input, the deserializer throws a domain-prefixed error that names the offending field but never echoes the field's value (an envelope's contents may include secrets).
- Forward-compatibility is preserved: extra unknown top-level fields continue to survive the deserialize round-trip exactly as they do today (the codex backup format is frozen at version 1.2 and this requirement does not change that).
- The Codex signing strategy gains a pre-flight check at the entry point of its execute path. When a transaction requires a foreign-key signer and the configured resolver does not implement the foreign-key request method, the strategy throws a precise error before reaching the universal sign helper. The error names the missing method and points the consumer at the resolver they configured.
- The JSDoc on the foreign-key request method in the resolver interface is clarified to state the runtime contract: the method is optional in the interface (server resolvers that never sign foreign-key transactions may omit it), but it is required at execute time the moment any guard requires a foreign key.

### Guard hardening and typed predicate error (M1 Phase 3)
- The guard-kind classifier requires the FULL minimal shape of each kind before classifying. A capability guard requires the capability name, arguments, and pact identifier all to be present. A user guard requires the function and arguments. A keyset guard requires the predicate and keys. A keyset-reference guard accepts either casing of the reference field (camelCase used internally and lowercase emitted by the chain). Under-specified payloads that previously slipped through and silently mis-classified now classify as `unknown` and are surfaced to the caller via the existing analysis surface.
- A normalization helper is applied at the resolve-guard boundary so that internal code only ever sees the camelCase reference field. The helper maps the lowercase chain-emitted variant to the camelCase form at the chain-IO boundary, which means the rest of the guard module no longer carries either-or branches.
- The Smart Ouronet Account auth-paths analysis result documents its four reachable states explicitly in JSDoc: a satisfiable path exists; no satisfiable path but at least one key-based path exists; no satisfiable path and no key-based path but at least one path of a known kind exists; all paths classify as unknown. An optional field is added to the analysis surface naming the index of the first signable-but-unsatisfied path so that consumers can render a "you would be able to sign if you held key X" UX without recomputing.
- The threshold-computation helper now throws a NEW typed error class (an additive public export from the guard subpath) when it encounters an unrecognized predicate. The general-purpose `analyzeGuard` function catches that typed class and folds the condition into a structured `predicateRecognized: false` bit on the returned analysis. The previous silent console diagnostic is removed; the typed error and the structured bit replace it.

### Documentation refresh (M2 Phase 4)
- The README header version table is refreshed from its current v1.3.0 / v1.4.0 baseline to current v2.2.0 reality, and cross-references the changelog for per-version detail rather than restating it.
- The CONTEXT document's interactions section is refreshed to describe v1.4 (the account-selector data shape gaining the public-key, sovereign, and governor fields), v1.5 (the DALOS curve re-exports and the Gen1 primitive factory plus the address-prefix-pair type), and v1.6 (the Smart Ouronet Account auth-path resolution primitives plus the rotate-sovereign Pact-code builder). The phase notes coordinate with the `/bee:refresh-context` command: if that command is run after subsequent code changes, the section may be redone wholesale and the manual refresh in this phase is acceptable as a snapshot of the current state.

### Catch-block consistency (M2 Phase 5)
- The catch blocks in the ouro-functions interaction file currently mix two conventions: silent at four sites and console-logging at three sites. After this phase, all seven sites apply ONE convention: route via the central logger seam's error method (locked iter 1 per spec-review I-004 — both phases land in the same atomic v2.3.0 commit, so the routed path is the higher-value outcome that closes F-CORE-019 cleanly without leaving the changelog text path-dependent). The convention is documented in a code comment near the affected handlers.
- The outer try/catch on the LP-type-info function in the add-liquidity interaction file is dropped. The outer catch is dead code today: the wrapped expression awaits a Promise.all of two promises that never reject, so the outer catch can never fire. Dropping the outer catch means that if a future contributor removes one of the inner catches, the now-broken behavior surfaces as a real failure rather than silently masking under the dead outer handler.

### Tier semantics and central logger seam (M2 Phase 6)
- The pact-reader and the raw calibrated reader gain JSDoc enumerating the canonical tier mapping (T1 for balance reads, T2 for preview reads, T3 for metadata reads, T7 for very-static reads, and so on) and clarifying that the default reader accepts and ignores the tier argument. The JSDoc cross-references the pact-reader injection seam for consumers who want a cache-aware reader. The optional development-mode warning on tier-less reads is dropped from this phase's scope (locked iter 1 per spec-review advisory #2) — JSDoc-only clarification is sufficient; runtime warnings are out of scope.
- A NEW central logger seam is introduced as the only NEW PUBLIC SURFACE in this release alongside the typed predicate error class. The seam mirrors the existing pact-reader injection pattern — module-private state, a setter, and a getter — and is exposed under a NEW `./observability` subpath in the package's exports map. The seam exports a `Logger` type carrying a warn method and an error method (each accepting a message string followed by additional unknown arguments), a `setLogger` mutator that swaps the active logger, and a `getLogger` accessor that returns the currently-configured logger.
- The default logger routes its warn calls to `console.warn` and its error calls to `console.error`, so consumers who do nothing observe behavior identical to the previous direct calls. Consumers who supply their own logger via the setter redirect every library log call into their sink. The setter rejects null or undefined input by throwing `TypeError` with message exactly `setLogger requires a non-null Logger` (LOCKED iter 1 per spec-review I-003; see REQ-13 for the requirements-level pin) rather than silently installing a non-functional logger.
- Every direct `console.warn` and `console.error` call in the source tree is replaced with a call routed through the logger accessor. After this phase, a search for direct console-warn or console-error calls in the source tree returns zero matches. The audit identified approximately twenty-five direct console call sites across roughly eight files; all are routed through the seam.

### Release artifacts (Phase 7)
- The package version field is bumped from 2.2.0 to 2.3.0.
- The changelog gains a new leading entry dated for the release. The format mirrors the v2.2.0 single-concern minor entry: a lead paragraph describing the additive nature of the release, an "Added (public surface)" section listing the new typed predicate error class and the new `./observability` subpath with its three exports, a "Fixed" section grouping the closed findings by milestone (M1: F-CORE-013/014/015/016a/b/c/017; M2: F-CORE-018a/b/019/020/021/022), and a "Stats" block citing the new test count and the files-changed list.
- The README is updated per the locked maintenance rule: the Status block leads with the new version and a one-line summary of the additive changes (closes 13 audit findings — 7 MEDIUM and 6 LOW; introduces the central logger seam at a new subpath); the version history is extended with a v2.3.0 paragraph; test-count references throughout the README are refreshed to the post-phase actual count; a NEW submodule table row documents the `./observability` subpath; an optional "what's new in v2.3.0" section may include a copy-paste example showing how a consumer routes log events via the new seam (mirroring the v2.1.0 / v2.2.0 precedent).
- The publish workflow's three-check version-parity gate is satisfied by the README and changelog updates so that the publish step does not block the annotated tag. The three checks must reference the new version byte-identically to the locked grep patterns (the workflow already enforces this — the requirement is that the README anchors and the changelog leading heading match the locked anchors so the gate passes).
- A verification gate runs at the end of the phase: typecheck must exit zero; the test suite must exit zero (the pre-existing Windows non-en-US locale-sensitive gas-format test continues to be the sole allowed non-zero exit, matching the criterion locked in v2.1.2 and v2.2.0); the build must exit zero; the new `./observability` subpath's compiled artifacts must be present in the published distribution; a runtime export verification confirms the three new exports load via a dynamic ESM import.

### Atomic-ship and additive constraints (cross-cutting)
- All public-API additions are ADDITIVE. No symbol renames; no return-type widening; no public exports removed; no behavior change on any path that was previously reachable with valid input. The only newly-thrown errors fire on inputs that were previously broken in different, less-clear ways. Strict semver is preserved: this is a minor bump (v2.2.0 → v2.3.0), and consumers upgrade transparently.
- The type-regression lock established in v1.7.0 continues to pass at every phase boundary.
- The combined CI test wall-time stays under 30 seconds on Node 22 / Ubuntu, comfortably above the current ~15-20s baseline.
- No new "Open handles" warnings from the test runner are introduced by any phase.
- Phase 1 produces zero test count change (mechanical refactor, byte-identical behavior). Each of Phase 2, Phase 3, and Phase 6 adds at least three new test cases covering the new contract surface (shape validation cases, foreign-key pre-flight pre-condition, predicate-recognition flag, logger seam null-rejection and getter/setter contract).
- All 7 phases land in a SINGLE commit and a SINGLE annotated v2.3.0 tag. Single npm publish; single GitHub Release covers the entire spec.

## Existing Code to Leverage
- The pact-reader injection seam (module-private state plus `set`/`get` accessor pair) is the direct template for the new central logger seam. Mirror its module-private state, accessor-pair shape, and JSDoc style. The default-routing-to-console behavior parallels the pact-reader's default-to-raw-calibrated-dirty-read behavior.
- The v2.2.0 release precedent is the closest template for: workflow-gate verbatim grep patterns; the changelog bold "**N** passing" convention; the README placeholder pattern for test-count refresh; the verification-gate structure run at the end of the release phase; per-task wall-time budgets in the verification step.
- The crypto barrel's existing pattern of re-exporting three error classes from a sibling errors file is the template for the new typed predicate error class re-export from the guard subpath.
- The encryption test file's existing console-spy pattern (introduced during v2.2.0) is the template for verifying that the logger seam catches every previously-direct console call site.
- The canonical `safeCreationTime` declaration in the pact-format module is the single source of truth for the Phase 1 deduplication. The eleven duplicate declarations in the interaction files are byte-identical to it; the refactor replaces each with an import.
- The codex codec's frozen v1.2 envelope contract (with its locked unknown-extra-field forward-compat behavior) is preserved unchanged by Phase 2's new shape checks. The new validation runs after the existing version check and before the cast.
- The Smart Ouronet Account auth surface and its `enforce-one`-style three-branch resolution (account guard, sovereign guard, governor guard) is preserved unchanged. Phase 3's hardening tightens classification and adds a structured analysis bit; the underlying signing-strategy contract (single AND-of-keysets array chosen by the consumer) is untouched.

## Out of Scope
- F-CORE-007 (fabricated fallback values in three read helpers — return-type widening, BREAKING) is deferred to a separate v3.0.0 major spec (M3 of the unified bundle). Closing it requires a major bump that consumers must adopt deliberately; bundling it with the additive M1+M2 work would force a major bump on a release that is otherwise transparent.
- The catalog sweep of approximately twenty-two additional empty-catch sites across the interactions module beyond the three representative functions covered by F-CORE-007 is also out of scope here. It is logged as an optional secondary task in the v3.0.0 spec.
- F-CORE-009/011/012 (closed in v2.2.0), F-CORE-008 (closed in v2.1.0), F-CORE-002/003/004 (closed in v2.1.0), and F-BUG-001 (closed in v2.1.2) are not revisited.
- No `AUDIT.md` artifact is created. The changelog continues to be the canonical record of closed findings; the audit-spec sources under `.bee/audit-specs-unified/` retain the per-finding evidence trail.
- No new runtime dependencies. All phases use existing imports and existing patterns. The new logger seam ships as part of the existing TypeScript build pipeline; no new tooling is introduced.
