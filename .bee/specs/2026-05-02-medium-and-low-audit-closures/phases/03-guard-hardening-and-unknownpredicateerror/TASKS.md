## Goal

Tighten four guard-related contracts in the guard subpath, and introduce ONE new typed error class as the FIRST of two new public surfaces in the v2.3.0 release:

1. **REQ-04 (F-CORE-016a) -- `classifyGuardKind` minimal-shape tightening.** `classifyGuardKind` at `src/guard/smartAccountAuth.ts:84-106` currently returns `keyset`/`keyset-ref`/`capability`/`user` for under-specified payloads (e.g. an object with only `cgName` and no `cgArgs`/`cgPactId` classifies as `capability`). Phase 3 tightens the discriminator to require the FULL minimal shape per kind: `capability` needs `cgName` (string) AND `cgArgs` (defined) AND `cgPactId` (defined, may be `null` per Pact convention); `user` needs `fun` (string) AND `args` (defined); `keyset` needs `pred` (string) AND `keys` (array); `keyset-ref` accepts EITHER casing of the ref field (camelCase `keysetRef` used internally OR lowercase `keysetref` emitted by the chain). Under-specified payloads classify as `unknown` instead of silently mis-classifying. Surfaces the ambiguity to the caller through the existing `GuardKind` union -- no new analysis-surface field for this requirement.

2. **REQ-05 (F-CORE-016b) -- `normalizeKeysetRef` boundary helper.** A new internal helper `normalizeKeysetRef(g)` at `src/guard/smartAccountAuth.ts` (added near `extractKeysetFromGuard` at lines 118-126) maps the lowercase chain-native `keysetref` field to camelCase `keysetRef`. Applied at the `resolveGuard` boundary inside `src/interactions/ouroFunctions.ts:191-216` (the chain-IO boundary) so internal code only ever sees camelCase. The rest of the guard module no longer carries either-or branches on the casing. The helper is NOT exported from the guard subpath barrel (it is an internal-to-the-guard-module concern); `classifyGuardKind`'s `keyset-ref` branch keeps the dual-casing acceptance for defense-in-depth in case a caller bypasses the boundary.

3. **REQ-06 (F-CORE-016c) -- `SmartAccountAuthPaths` JSDoc + new optional field.** The `SmartAccountAuthPaths` interface at `src/guard/smartAccountAuth.ts:171-179` and the JSDoc above it at lines 153-170 are extended: the JSDoc now ENUMERATES the four reachable states explicitly (state A: `firstSatisfied >= 0` -- a satisfiable path exists; state B: `firstSatisfied === -1 && anyKeyBased === true` -- no satisfiable path but at least one key-based path exists, the consumer can pick a branch and prompt for the missing key; state C: `firstSatisfied === -1 && anyKeyBased === false && at least one branch.kind !== "unknown"` -- no signable path but at least one path is of a known kind, the consumer should refer the user to Execute Code; state D: all branches classify as `unknown` -- the analyzer cannot help). The interface gains an OPTIONAL field `firstSignableButUnsatisfied?: number` (the index of the first branch whose `keyBased === true` AND `analysis.satisfied === false` -- supports the "you would be able to sign if you held key X" UX without a recompute). The field is `-1` (or `undefined` -- implementer chooses one and pins it in the JSDoc) when no such branch exists; the canonical value is locked by the implementer's choice and asserted in the new tests.

4. **REQ-07 (F-CORE-017) -- `UnknownPredicateError` typed class + `predicateRecognized` analysis bit.** `computeThreshold` at `src/guard/guardUtils.ts:76-79` currently `console.warn`s and falls back to `keyCount` (i.e. `keys-all` semantics) when it encounters an unrecognized predicate. Phase 3 replaces that silent diagnostic with a typed `UnknownPredicateError` (NEW class, additive public export from the guard subpath -- LOCKED I-002 NOT errors subpath) thrown by `computeThreshold`. `analyzeGuard` at `src/guard/guardUtils.ts:119-157` catches the typed class and folds it into a NEW structured `predicateRecognized: boolean` field on the returned `GuardAnalysis` (true on success, false on caught `UnknownPredicateError` -- which means the threshold defaults to `keyCount` / keys-all and the caller can branch on the bit cleanly). `predicateLabel` at `src/guard/guardUtils.ts:83-89` calls `computeThreshold` and would propagate the throw -- it MUST be updated to either catch the typed class or pre-check the predicate so it does not throw on unknown predicates (preserves the existing test at `tests/guard.test.ts:133-137` which calls `predicateLabel` for known predicates only -- but defensively we want `predicateLabel` to remain non-throwing for unknown preds since it is a UI helper). The previous silent `console.warn` is removed.

5. **At least 3 new test cases** (per NFR-04 minimum-3-new-tests-per-substantive-phase) -- realistically 6-8 given the four sub-findings: (i) under-specified `capability` shape classifies as `unknown`; (ii) under-specified `user` shape classifies as `unknown`; (iii) under-specified `keyset` shape classifies as `unknown`; (iv) lowercase `keysetref` normalizes to camelCase via the new helper; (v) `firstSignableButUnsatisfied` returns the correct index in state B; (vi) `computeThreshold` on unknown predicate throws `UnknownPredicateError`; (vii) `analyzeGuard` on unknown predicate sets `predicateRecognized: false` AND falls back to keys-all; (viii) `UnknownPredicateError` is re-exported from `@stoachain/ouronet-core/guard` and is `instanceof Error`. Test files: extend existing `tests/guard.test.ts` for `computeThreshold` / `analyzeGuard` / `UnknownPredicateError` cases; extend existing `tests/smart-account-auth.test.ts` for `classifyGuardKind` tightening, the `normalizeKeysetRef` helper round-trip via `analyzeSmartAccountAuthPaths`, and `firstSignableButUnsatisfied`.

6. **Existing-test regression handling.** `tests/guard.test.ts:115-119` (`unknown predicate falls back to keys-all (conservative)`) currently asserts `computeThreshold("some-unknown-predicate", 5)` returns `5`. Under REQ-07 this test MUST be UPDATED to assert the new throw behavior, OR REPLACED by a sibling test that asserts the throw.
