## 1. Background

OuronetUI has, since pre-v1.0, treated the codex as having two unremovable identity anchors:

- **Prime Codex Seed** — the very first `IKadenaSeed` the codex was kickstarted with. Removing it = removing the codex.
- **CodexPrime** — the very first `IOuroAccount`, a Standard Ouronet Account (non-Smart) derived from the SAME mnemonic as the Prime Codex Seed.

The v0.1.0 package partially implemented this: `IOuroAccount.isPrime` exists, `addOuroAccount` auto-flags the first added account, and `deleteOuroAccount` refuses to delete prime accounts. But three gaps remain that break the invariant under any consumer other than OuronetUI's hand-orchestrated `saveWallet()`:

1. No symmetric `IKadenaSeed.isPrime` — the Prime Codex Seed has zero deletion protection.
2. No `kickstartCodex` action — consumers must manually call `addKadenaSeed` + `addOuroAccount` in sequence, and any consumer doing it in the wrong order (or skipping a step) breaks the invariant.
3. No identity binding — `isPrime: true` on an ouro account is positional ("first added wins"), not causal ("derived from the prime seed").

v0.2.0 closes all three gaps. Changes are **additive** — v0.1.0 consumers continue to work unchanged; the new surface is the preferred path forward.

## 2. Goals

- The Prime Codex Seed and CodexPrime account are **structural guarantees** of every codex, not conventions enforced by each consumer.
- A consumer implementing a new app (e.g. AncientHoldings hub) can call ONE function to create a fresh codex correctly. No surface for getting the invariant wrong.
- Existing v0.1.0 codexes on disk migrate cleanly on next load — no user-visible action required.
- Backward compat: every public symbol from v0.1.0 keeps its signature. New behavior is additive.

## 3. Non-goals

- **No key derivation in the package.** DALOS / kadena HD key derivation stays in stoa-core. The package takes pre-derived entities as input, owns the invariants over them, and orchestrates persistence. Consumers continue to use `generateOuronetAccount` + `KadenaWalletBuilder.createWalletPairFromMnemonic` from upstream.
- **No "codex = mnemonic" assumption.** A codex CAN contain multiple kadena seeds (only the first is prime), pure keypairs (zero relation to seeds), and ouro accounts derived from different seeds. The prime invariants only constrain the FIRST seed and ITS derived ouro.
- **No retroactive enforcement of `parentSeedId` on legacy ouro accounts.** Best-effort migration; legacy accounts that don't match any seed stay unbound.

## 4. Type changes (additive)

### 4.1 `IKadenaSeed` — add `isPrime`

```typescript
export interface IKadenaSeed {
  id: string;
  name?: string;
  seedType: SeedType;
  version: string;
  index: number;
  secret: string;
  main: string;
  createdAt: string;
  accounts: WalletAccount[];

  /** Prime Codex Seed marker. True for the seed that kickstarted the
   *  codex; false (or absent) for additional seeds added later. The
   *  package's `deleteKadenaSeed` throws `CodexPrimeSeedProtectedError`
   *  on `isPrime: true` seeds. There is exactly one prime seed per
   *  codex, set atomically by `kickstartCodex`. */
  isPrime?: boolean;
}
```

### 4.2 `IOuroAccount` — add `parentSeedId`

```typescript
export interface IOuroAccount {
  // ... existing fields ...
  isPrime?: boolean;

  /** ID of the IKadenaSeed this account was derived from. Undefined for
   *  pure-keypair-derived accounts (which have no parent seed). Set by
   *  `kickstartCodex` on the CodexPrime account so the prime-account
   *  invariant is causal ("derived from the prime seed") rather than
   *  positional ("first added"). */
  parentSeedId?: string;
}
```

## 5. New API surface

### 5.1 `CodexPrimeSeedProtectedError`

```typescript
export class CodexPrimeSeedProtectedError extends CodexError {
  public override readonly name = "CodexPrimeSeedProtectedError";
  public readonly seedId: string;

  constructor(seedId: string) {
    super(
      `Kadena seed ${seedId} is the Prime Codex Seed and cannot be ` +
        `deleted. Removing the prime seed is equivalent to deleting the ` +
        `codex itself — use the codex-reset flow instead.`
    );
    this.seedId = seedId;
  }
}
```

### 5.2 `kickstartCodex` action

```typescript
interface KickstartArgs {
  /** Pre-formed kadena seed. Caller has already derived keypairs from the
   *  mnemonic + encrypted the secret at the codex password. */
  seed: Omit<IKadenaSeed, "isPrime">;
  /** Pre-formed ouro account. MUST be a Standard Ouronet Account
   *  (isSmart: false) derived from the SAME mnemonic as `seed`. Package
   *  asserts isSmart === false; throws otherwise. */
  primeOuroAccount: Omit<IOuroAccount, "isPrime" | "parentSeedId">;
}

interface KickstartResult {
  seed: IKadenaSeed;       // with isPrime: true
  primeOuro: IOuroAccount; // with isPrime: true, parentSeedId: seed.id
}

// On the actions interface:
kickstartCodex(args: KickstartArgs): Promise<KickstartResult>;
```

**Refuses (throws CodexKickstartError) when:**
- The codex already contains any kadena seed (use `addKadenaSeed` for additional seeds).
- `primeOuroAccount.isSmart === true` (CodexPrime must be a Standard account).

**Effect:** atomically persists the seed (with `isPrime: true`) AND the ouro (with `isPrime: true` + `parentSeedId: seed.id`). Sets `activeKadenaWalletId` and `activeOuroAccountId` to the new entities. Fires one `dirty: true` transition, not two.

### 5.3 `recoverCodexFromMnemonic` action

```typescript
recoverCodexFromMnemonic(args: KickstartArgs): Promise<KickstartResult>;
```

Same signature as `kickstartCodex` but for the recovery flow. Differs in two ways:
1. Allows non-empty codex (overwrites existing prime seed/account if present, after refusing if their IDs differ — see §6.3).
2. Does NOT auto-activate (caller decides activation policy).

In practice, both `kickstartCodex` and `recoverCodexFromMnemonic` share an internal `_installPrimePair(args)` helper; only the pre-flight guards differ.

### 5.4 New error class

```typescript
export class CodexKickstartError extends CodexError {
  public override readonly name = "CodexKickstartError";
  public readonly reason: "already-kickstarted" | "smart-account-not-allowed" | "id-conflict";
}
```

## 6. Semantics

### 6.1 Auto-prime-on-first-add (existing behavior, refined)

`addKadenaSeed`:
- If codex has zero seeds AND `seed.isPrime` is unset → auto-set `isPrime: true`.
- If codex already has any seed → leave `isPrime` as caller provided (defaults to false/absent).
- **NEW guard:** if caller passes `isPrime: true` AND a prime seed already exists, throw `CodexKickstartError("id-conflict")`. Exactly one prime seed per codex.

`addOuroAccount`:
- Existing behavior preserved: first-added auto-flags `isPrime: true`.
- **NEW refinement:** if caller provides `parentSeedId`, the package verifies it matches an existing seed; if no match, the field is dropped (warning logged). This guards against typos but does not require the seed to exist (consumer might be importing an orphaned account from a backup).
- **NEW guard:** if caller passes `isPrime: true` AND a prime ouro already exists, throw `CodexKickstartError("id-conflict")`.

### 6.2 Deletion protection

`deleteKadenaSeed(id)`:
- If target.isPrime → throw `CodexPrimeSeedProtectedError(id)`.
- Else: standard delete. **Cascade**: any ouro accounts whose `parentSeedId === id` are also deleted (this matches the user expectation — deleting a non-prime seed should not leave orphaned ouro accounts that can no longer sign). Cascade does NOT trigger `CodexPrimeProtectedError` even if the ouro had `isPrime: true` (which it shouldn't, because primes are tied to the prime seed, but defensive). Log a warning if it happens.

`deleteOuroAccount(id)` — unchanged (already throws CodexPrimeProtectedError on isPrime).

### 6.3 Recovery semantics

`recoverCodexFromMnemonic`:
- If codex is empty → identical to kickstart.
- If codex has a prime seed already AND `args.seed.id === existingPrimeSeed.id` → idempotent re-install (overwrites fields).
- If codex has a prime seed already AND `args.seed.id !== existingPrimeSeed.id` → throw `CodexKickstartError("id-conflict")`. Caller must reset codex first or generate `seed.id` to match.

### 6.4 Backup/restore (v1.2 codex file format)

The v1.2 backup format is FROZEN (per stoa-js CLAUDE.md). The `isPrime` and `parentSeedId` fields are passed through as additional properties on the persisted entities — older codex parsers ignore unknown fields (additive JSON convention), and `useCodexBackup().restore()` round-trips them.

On `restore`, if the imported snapshot has zero seeds with `isPrime: true` but at least one seed exists, run the §8 migration in-line (auto-flag the first seed + matching ouro). This ensures backups from v0.1.0 → restored into v0.2.0+ become fully compliant.

## 8. Migration of pre-v0.2.0 codexes on disk

In `actions.init(adapter)`, after `adapter.loadAll()`:

```typescript
// Detect: codex has at least one seed but no seed has isPrime.
const needsMigration = snap.kadenaSeeds.length > 0 &&
                       !snap.kadenaSeeds.some(s => s.isPrime);

if (needsMigration) {
  const primeSeed = snap.kadenaSeeds[0];
  primeSeed.isPrime = true;

  // Find the ouro whose encrypted secret decrypts to the same mnemonic.
  // BLOCKED: this requires the codex password, which the user hasn't
  // entered yet at init-time. RESOLUTION: defer ouro flagging until
  // authenticate(password) succeeds — there's already a natural hook
  // there. Until then, kadenaSeed flagging is done eagerly.

  // Persist seed update immediately (no password needed).
  await adapter.saveKadenaSeeds(snap.kadenaSeeds);
}
```

**Deferred half**: in `actions.authenticate(password)`, after successful
password unlock, run a one-time check: if any seed has `isPrime: true`
but no ouro has `parentSeedId` matching it, attempt to find the matching
ouro by decrypting each candidate's `secret` field with the now-known
password and comparing. The first ouro whose secret decrypts to a
mnemonic that, when re-fed to the caller's key-derivation, matches the
prime seed's `main` publicKey gets flagged.

Catch: this requires the caller to provide a derivation function. v0.2.0
exposes an OPTIONAL `actions._migrateLegacyOuroFlags(deriveFn)` that the
consumer can invoke once at init. If consumer doesn't call it, the ouro
half of the migration is skipped — but the seed half (which is the more
important protection) still applies eagerly.

**Pragmatic alternative considered + rejected**: auto-flag `ouroAccounts[0]`
as prime on load (without verifying derivation). Rejected because it
breaks the "causal, not positional" rule established for v0.2.0 — if a
user ever reordered accounts (no UI for this today, but defensive), the
wrong account would be permanently locked as undeletable. Migration must
be best-effort + causal, not lazy + positional.
