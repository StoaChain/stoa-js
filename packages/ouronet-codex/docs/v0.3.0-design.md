# @stoachain/ouronet-codex v0.3.0 — Design Lock

**Status:** LOCKED 2026-05-29 — ready for implementation
**Authors:** Mihai (owner) + Claude
**Drives:** the first ouronet-ns.CODEX on-chain registration of the Mnemosyne admin Codex, which unblocks the rest of Mnemosyne v0.1
**Reference:** `Mnemosyne/docs/v0.1-design.md` (architecture context) and `Mnemosyne/docs/handoffs/01-mnemosyne-codex-pact-module.md` (the on-chain side)

All changes in this version are **additive** to v0.2.0 — every v0.2.0 export keeps its signature. Existing codices auto-migrate on first load via the new schema-migration framework. No breaking changes.

---

## 1. Why v0.3.0 happens — the goal

v0.3.0 ships exactly what's needed to:

1. **Generate** a Codex with the new on-chain-aware structure (Codex Identity, CodexGuard, optional Duo Pure Prime, namespaced Consumer Settings)
2. **Build** the unsigned Pact transaction that registers the Codex Identity on the on-chain `ouronet-ns.CODEX` module
3. **Run** on Node.js (the Mnemosyne backend) as well as in the browser (existing consumers: OuronetUI, AncientHoldings hub)
4. **Migrate** existing v0.2 codices to v0.3 shape on first load, preserving every byte of user data

That's the entire scope. Everything not strictly required for those four goals is deferred to v0.3.x.

## 3. Type changes

### 3.1 `ICodexIdentity` — new

Every codex created or upgraded under v0.3.0 has exactly one Codex Identity. It is **IMMUTABLE** post-creation (the on-chain registration depends on this).

```typescript
export interface ICodexIdentity {
  /** Display-only formatted address.
   *  Format: ₱.STANDARD-payload:Π.SMART-payload
   *  Where each payload is 160 glyphs from the Dalos character set encoding
   *  the corresponding Apollo public key. The ':' between halves matches the
   *  styling of the prefix dots so the whole identifier reads cohesively. */
  formatted: string;

  /** Apollo Standard public key (160-glyph Dalos encoding of 1024 bits). */
  standardPublicKey: string;

  /** Apollo Smart public key (160-glyph Dalos encoding of 1024 bits). */
  smartPublicKey: string;

  // ─── Encrypted-at-CK private material ─────────────────────────────────
  // ALL of these are derivable from `encryptedSeedWords` alone; the other
  // representations are cached for two reasons:
  //   1. Avoid re-running HKDF/PBKDF derivation on every sign
  //   2. Let the user export the seed in whichever format they want without
  //      re-running derivation in the SPA

  /** Full seed word sequence (1-512 words, 1-256 glyphs/word from the Dalos
   *  character set). UTF-8 string, space-separated. */
  encryptedSeedWords: string;

  /** 1024-bit Standard half as a binary string. */
  encryptedStandardBitstring: string;

  /** 1024-bit Smart half as a binary string. */
  encryptedSmartBitstring: string;

  /** Standard half as a base-10 scalar (up to ~309 decimal digits). */
  encryptedStandardBase10: string;

  /** Smart half as a base-10 scalar. */
  encryptedSmartBase10: string;

  /** Standard half as a base-49 scalar (Apollo's natural base). */
  encryptedStandardBase49: string;

  /** Smart half as a base-49 scalar. */
  encryptedSmartBase49: string;

  /** Optional cached Apollo private keys (re-derivable from bitstrings).
   *  Populated to avoid Apollo-curve derivation on every sign operation. */
  encryptedStandardPrivateKey?: string;
  encryptedSmartPrivateKey?: string;

  // ─── Plaintext metadata ───────────────────────────────────────────────

  /** Total seed-word count, 1 to 512.
   *  Combined with splitIndex this reconstructs the half boundaries. */
  totalWordCount: number;

  /** The index where the Smart half starts.
   *  Rule (deterministic, no user override at v0.3.0):
   *    splitIndex = Math.floor(totalWordCount / 2)
   *  So:
   *    totalWordCount=6  → splitIndex=3 → Standard=3 words, Smart=3 words
   *    totalWordCount=7  → splitIndex=3 → Standard=3 words, Smart=4 words (Smart bigger)
   *    totalWordCount=11 → splitIndex=5 → Standard=5 words, Smart=6 words (Smart bigger)
   *  Smart half ALWAYS gets the larger share when word count is odd. */
  splitIndex: number;

  createdAt: string;

  /** Mnemosyne account username that triggered this Codex's creation.
   *  Empty string for codices created outside Mnemosyne (e.g. via OuronetUI
   *  directly). Stored for audit; not used for authorization. */
  createdBy?: string;
}

// On CodexSnapshot:
export interface CodexSnapshot {
  // ... existing v0.2.0 fields ...
  codexIdentity?: ICodexIdentity;  // exactly one per codex, IMMUTABLE post-creation
}
```

### 3.2 `IPureKeypair` — extend with CodexGuard + DuoPurePrime markers

```typescript
export interface IPureKeypair {
  // ... existing v0.2.0 fields (id, label?, publicKey, encryptedPrivateKey, createdAt) ...

  /** Active CodexGuard marker.
   *  Exactly ONE pure key in the codex carries isCodexGuard: true at any time.
   *  That key's properties:
   *    - is the FIRST pure key created when the codex was kickstarted
   *    - has its label LOCKED to "CodexGuard" (codex rejects rename attempts)
   *    - is NEVER deletable (codex rejects delete attempts)
   *    - UI renders with the dark-cherry designation color (consumer-side hint
   *      to display this distinctively)
   *    - is the key whose public half is registered as the codex-guard field
   *      on the ouronet-ns.CODEX chain entry
   *  Set by kickstartCodex on creation; transferred by rotateCodexGuard. */
  isCodexGuard?: boolean;

  /** Historical CodexGuard marker.
   *  When the user rotates the CodexGuard (replaces with a new pure key):
   *    - the NEW pure key gets isCodexGuard: true (and isCodexGuard is
   *      cleared from the prior one)
   *    - the OLD pure key gets wasCodexGuard: true
   *    - the OLD pure key's delete-protection STAYS (it remains stored
   *      forever as a "former CodexGuard")
   *    - the OLD pure key's label remains "CodexGuard" but with a numeric
   *      suffix indicating retirement order: "CodexGuard (retired #1)"
   *      (rename guards relax to allow this controlled suffix; consumer UI
   *      makes the historical nature clear)
   *  This preserves the historical chain of CodexGuards forever — anyone
   *  inspecting the codex can see every key that ever held this role. */
  wasCodexGuard?: boolean;

  /** Duo Pure Prime marker (autopilot CodexPrime backing).
   *  When the user chooses "auto-pure-keys" mode at kickstart instead of
   *  providing a Kadena seed, the package generates two pure keys to back
   *  the CodexPrime Ouronet Account (one as Payment Key, one as the
   *  Ouronet account's keyset key). Both keys get isDuoPurePrime: true
   *  and are protected from deletion (same mechanism as CodexGuard).
   *  Their labels are locked to "Duo Pure Prime (Payment)" and
   *  "Duo Pure Prime (Guard)" respectively. */
  isDuoPurePrime?: boolean;

  /** Role within the Duo Pure Prime pair (only meaningful when
   *  isDuoPurePrime: true). */
  duoPurePrimeRole?: "payment" | "guard";
}
```

### 3.3 `IConsumerSettings` — new + `CodexSnapshot.consumerSettings`

```typescript
/** Per-consumer settings entry. Each consumer (OuronetUI, AncientHoldings,
 *  Mnemosyne, future apps) owns ONE entry keyed by its canonical name and
 *  writes ONLY to its own entry. Other entries are preserved verbatim on
 *  save (additive convention — never drop unknown consumer entries). */
export interface IConsumerSettings {
  /** Canonical consumer identifier (PascalCase, no spaces).
   *  Known consumers as of v0.3.0:
   *    - "OuronetUI"          (browser wallet SPA)
   *    - "AncientHoldingsHUB" (Node.js admin server)
   *    - "Mnemosyne"          (Node.js codex-vault backend)
   *  New consumers register their name and start writing to a fresh slot. */
  consumerName: string;

  /** Semver version of the consumer that last wrote these settings. */
  consumerVersion: string;

  /** Schema version internal to THIS consumer's settings shape. Bumped when
   *  the consumer's settings interface evolves. Each consumer manages its
   *  own schema version independently. */
  schemaVersion: number;

  /** Opaque-to-other-consumers settings blob, typed by the consumer.
   *  Other consumers MUST preserve this verbatim on save. */
  settings: Record<string, unknown>;

  /** ISO timestamp of last write. */
  lastUpdatedAt: string;
}

// On CodexSnapshot:
export interface CodexSnapshot {
  // ... existing fields ...
  consumerSettings?: Record<string, IConsumerSettings>;
}
```

**Migration of existing OuronetUI fields:** the v0.2 → v0.3 schema migration relocates any OuronetUI-specific keys currently stored in `UiSettings[extra: string]: unknown` into `consumerSettings["OuronetUI"].settings`. Generic codex-wide settings (e.g., `passwordCacheMinutes`) remain in `UiSettings`. Full cleanup of `UiSettings` happens in v0.3.1.

## 4. New API surface

### 4.1 New error classes

```typescript
/** Thrown when ICodexIdentity operations fail (creation, derivation,
 *  attempt to mutate an immutable field). */
export class CodexIdentityError extends CodexError {
  public override readonly name = "CodexIdentityError";
  public readonly reason:
    | "already-exists"      // tried to create a Codex Identity on a codex that already has one
    | "seed-invalid"        // seed material doesn't conform to (1-512 words, 1-256 glyphs each)
    | "seed-word-count"     // wordCount outside 1..512
    | "split-invalid"       // splitIndex outside [0, totalWordCount]
    | "immutable-field"     // attempt to mutate any field after creation
    | "missing-codex-identity"; // operation requires one and the codex doesn't have it
}

/** Thrown when CodexGuard operations fail. */
export class CodexGuardError extends CodexError {
  public override readonly name = "CodexGuardError";
  public readonly reason:
    | "already-exists"        // tried to create a CodexGuard when one is active
    | "missing-codex-guard"   // operation requires the active CodexGuard and there isn't one
    | "rename-rejected"       // attempt to rename a CodexGuard or wasCodexGuard pure key
    | "delete-rejected"       // attempt to delete a CodexGuard or wasCodexGuard pure key
    | "rotation-invalid";     // rotation target keypair is invalid (e.g., already CodexGuard)
}

/** Thrown when consumer settings operations fail. */
export class CodexConsumerSettingsError extends CodexError {
  public override readonly name = "CodexConsumerSettingsError";
  public readonly reason:
    | "invalid-consumer-name"  // name doesn't conform to PascalCase rules
    | "schema-downgrade"       // attempt to write at a lower schemaVersion than already stored
    | "missing-entry";         // tried to read a consumer that has no entry
}

/** Thrown when schema migration fails. */
export class CodexMigrationError extends CodexError {
  public override readonly name = "CodexMigrationError";
  public readonly reason:
    | "unknown-schema-version"  // codex schemaVersion is newer than this package supports
    | "migration-failed"        // a migration function threw
    | "post-condition-failed";  // migrated snapshot doesn't satisfy the new schema's invariants
}
```

### 4.2 `kickstartCodex` updated signature

Replaces the v0.2.0 `kickstartCodex` (whose old call pattern still works through a backward-compat adapter that defaults the new fields to autopilot).

```typescript
interface KickstartArgs {
  // ─── Step 1: Codex Identity seed ──────────────────────────────────
  /** The seed material from which the double-Apollo Codex Identity is
   *  derived. Mandatory. */
  codexIdSeed: {
    mode: "words" | "bitstring" | "bitmap" | "base10" | "base49";
    /** Encoded value per mode:
     *    words      → space-separated UTF-8 string (Dalos charset, 1-512 words, 1-256 glyphs each)
     *    bitstring  → "0"/"1" string, length 2048 (split in half by splitIndex)
     *    bitmap     → 32-row by 64-column or 45x46 grid encoded as a string
     *    base10     → decimal string up to ~616 digits
     *    base49     → base-49 string in Apollo's natural charset */
    value: string;
    /** For "words" mode only. Otherwise inferred. */
    wordCount?: number;
    /** Override the split point. Default: floor(wordCount/2) when mode==="words";
     *  fixed at 1024 (1024-bit halves) for other modes. */
    splitIndex?: number;
  };

  // ─── Step 2: CodexPrime Ouronet Account seed ──────────────────────
  /** How the Standard Ouronet account that becomes CodexPrime is seeded. */
  codexPrimeSeed:
    | { source: "reuse-codexid-whole" }      // valid only if codexIdSeed wordCount ≤ 256
    | { source: "reuse-codexid-standard" }    // first half of codexIdSeed
    | { source: "reuse-codexid-smart" }       // second half of codexIdSeed
    | { source: "fresh-dalos"; words: string };  // fresh Dalos seed, 1-256 words

  // ─── Step 3: How CodexPrime is backed by Stoa keys ────────────────
  /** CodexPrime is a Standard Ouronet Account that needs 2 StoaChain keys:
   *  one for Payment Key (k:account that pays gas), one for the Ouronet
   *  account's guard keyset. Either:
   *    - "kadena-seed":     provide a Kadena seed; positions #0 and #1 become
   *                          the two keys; the seed gets isPrime: true and
   *                          positions #0/#1 become unremovable
   *    - "auto-pure-keys":  package generates two pure Stoa keypairs and
   *                          marks both with isDuoPurePrime: true (unremovable) */
  duoPrime:
    | { mode: "kadena-seed"; seedType: "koala" | "chainweaver" | "eckowallet"; mnemonic: string }
    | { mode: "auto-pure-keys" };

  // ─── Optional metadata ────────────────────────────────────────────
  /** Mnemosyne account username that initiated the kickstart. Stored on
   *  ICodexIdentity.createdBy for audit. */
  createdByUsername?: string;
}

interface KickstartResult {
  /** Generated double-Apollo identity, fully cached + stored. */
  codexIdentity: ICodexIdentity;

  /** Generated CodexGuard pure keypair (FIRST pure key in the codex,
   *  isCodexGuard: true, label locked to "CodexGuard", undeletable). */
  codexGuard: IPureKeypair;

  /** The CodexPrime Standard Ouronet Account (isPrime: true). */
  codexPrime: IOuroAccount;

  /** Present iff duoPrime.mode === "kadena-seed".
   *  The provided Kadena seed marked as Prime Codex Seed (positions #0
   *  and #1 unremovable, seed itself unremovable). */
  primeCodexSeed?: IKadenaSeed;

  /** Present iff duoPrime.mode === "auto-pure-keys".
   *  The two pure keypairs backing CodexPrime, both with
   *  isDuoPurePrime: true and duoPurePrimeRole indicating which one is
   *  the Payment Key vs Guard Key. Both unremovable. */
  duoPurePrime?: [IPureKeypair, IPureKeypair];

  /** Present iff codexPrimeSeed.source === "fresh-dalos".
   *  The fresh Dalos seed that was generated for CodexPrime. */
  primeOuroAccountSeed?: IKadenaSeed;
}
```

The function performs all generation **atomically** — either every output entity is created and persisted, or the whole call fails and the codex is unchanged.

### 4.4 Extended deletion / mutation protections

The existing `deleteOuroAccount`, `deleteKadenaSeed`, and `deletePureKeypair` actions gain new throws to protect the new v0.3.0 entities:

| Attempt | Throws |
|---|---|
| Delete a CodexPrime Ouronet account (`isPrime: true`) | `CodexPrimeProtectedError` (already in v0.2.0) |
| Delete a Prime Codex Seed (`isPrime: true`) | `CodexPrimeSeedProtectedError` (already in v0.2.0) |
| Delete a CodexGuard pure key (`isCodexGuard: true` OR `wasCodexGuard: true`) | `CodexGuardError("delete-rejected")` (v0.3.0+) |
| Delete a Duo Pure Prime pure key (`isDuoPurePrime: true`) | `CodexGuardError("delete-rejected")` (v0.3.0+) — uses same error class with same reason for ergonomics; same protection mechanism |
| Rename a CodexGuard pure key | `CodexGuardError("rename-rejected")` (v0.3.0+) |
| Rename a Duo Pure Prime pure key | `CodexGuardError("rename-rejected")` (v0.3.0+) |
| Mutate any field on `ICodexIdentity` post-creation | `CodexIdentityError("immutable-field")` (v0.3.0+) |

The variable-strictness D2 rule from the design discussion: whether `primeCodexSeed` or `duoPurePrime` carries the unremovable status depends on which `duoPrime.mode` the user picked at kickstart. Only ONE of those two paths gets the protection per codex.

### 4.5 Updated `addOuroAccount` — preserve existing flags

When upserting an existing account, preserve all flags (`isPrime`, `parentSeedId`, any future-added flags) that the caller hasn't explicitly set in the new account object. Same defensive-merge pattern as v0.2.0's handling of `parentSeedId`.

## 5. Schema migration framework

```typescript
// In src/state/migrations.ts

export interface SchemaMigration {
  fromVersion: number;
  toVersion: number;
  /** Pure function — no I/O, no side effects. Idempotent on already-current
   *  snapshots (the runner only calls migrations whose fromVersion < current). */
  migrate: (snapshot: CodexSnapshot) => CodexSnapshot;
  description: string;
}

/** Registry of all schema migrations, applied in order on init() when the
 *  loaded snapshot's schemaVersion is older than the package's current
 *  version. Never delete past entries — old codices still need them. */
export const SCHEMA_MIGRATIONS: SchemaMigration[] = [
  // v0.3.0 migration: bring v0.2 codices to v0.3 shape
  {
    fromVersion: 1,
    toVersion: 2,
    description:
      "v0.3.0: initialize consumerSettings registry, generate CodexGuard if missing, " +
      "leave codexIdentity null (requires interactive user seed input to create)",
    migrate: (snap) => {
      // Initialize the consumer settings registry
      const consumerSettings = snap.consumerSettings ?? {};

      // Relocate any OuronetUI-specific keys from UiSettings.extra into
      // consumerSettings["OuronetUI"]. Detection: any UiSettings field not
      // in the generic-known set (see KNOWN_GENERIC_UI_SETTINGS) gets moved.
      // [Implementation deferred to migration coding task.]

      // codexIdentity stays null — generation needs the user's seed input.
      // The Mnemosyne SPA (or any v0.3-aware consumer) detects null and
      // prompts the user to create a Codex Identity at the next interactive
      // moment.
      return {
        ...snap,
        consumerSettings,
        codexIdentity: snap.codexIdentity ?? null,
      };
    },
  },

  // CodexGuard generation is NOT done in this passive migration — it requires
  // crypto.random and consistent timing, so we mark it for the interactive
  // generateCodexGuardForLegacy() flow instead. The migration just ensures
  // the field shape is consistent; actual key generation happens on first
  // call to generateCodexGuardForLegacy() when the consumer has user attention.
];

/** Current schema version this package writes. Bumped per migration. */
export const CURRENT_SCHEMA_VERSION = 2;

/** Returns true if the consumer's package version can safely WRITE a codex
 *  at the given loaded schema version. Returns false if the loaded codex is
 *  at a NEWER schema (writing would risk dropping unknown fields). */
export function canConsumerWrite(loadedSchemaVersion: number): boolean {
  return loadedSchemaVersion <= CURRENT_SCHEMA_VERSION;
}
```

### `actions.init(adapter)` extension

Extends the existing init flow to run migrations:

```typescript
async init(adapter, deviceVariant) {
  // ... existing v0.2.0 init logic (load snapshot, run legacy migrations) ...

  const snap = await adapter.loadAll();

  // v0.2.0 legacy migration (Prime Codex Seed flagging) — unchanged

  // v0.3.0 NEW: run schema migrations
  let migrated = applyMigrations(snap, SCHEMA_MIGRATIONS, CURRENT_SCHEMA_VERSION);
  if (migrated.schemaVersion !== snap.schemaVersion) {
    // Persist eagerly so the upgrade sticks
    try { await adapter.saveAll(migrated); } catch { /* best-effort */ }
  }

  set({ /* ... use migrated instead of snap ... */ });
}
```

## 6. Node.js + browser parity

v0.3.0 must run on both Node.js (Mnemosyne backend) and browser (OuronetUI). The existing encryption + crypto paths already work in both — the package uses `@noble/hashes` and standard WebCrypto-compatible primitives. Specific items to verify during implementation:

| Dependency | Browser-safe? | Node-safe? | Action |
|---|---|---|---|
| Existing AES-GCM encrypt/decrypt | ✓ (WebCrypto) | ✓ (built-in node crypto) | No change |
| `@noble/hashes` for Blake3 | ✓ | ✓ | No change |
| Apollo-curve derivation via dalos-crypto | ✓ | ✓ | No change |
| Pact tx construction via `@kadena/client` | ✓ | ✓ | Already isomorphic |
| File-system codex adapter (new for Mnemosyne backend) | N/A | new | Add `FileSystemCodexAdapter` as a new adapter alongside `LocalStorageCodexAdapter` + `MemoryCodexAdapter`. v0.3.0 ships the type; Mnemosyne can implement its own variant if needed. |

## 10. Out of scope — explicit deferrals

These are deliberately NOT in v0.3.0 even though they appear in the broader Mnemosyne design context:

- **All React hooks** (`useGuardians`, `useHerotag`, `useConsumerSettings`, `useArweaveBackup`). Deferred to v0.3.1 when OuronetUI integrates v0.3. The Mnemosyne backend uses store actions directly.
- **Arweave backup actions** (`exportEncryptedSnapshotForArweave`, `buildUpdateArweavePointerTx`, `recordArweaveBackupLocally`, `refreshArweaveBackupHistory`, `getLatestArweaveBackupTxId`). Deferred to v0.3.1. Mnemosyne v0.1 launches with L1 (DB) + L3 (chain) only; L2 (Arweave) becomes opt-in later. The action surface leaves an "Arweave actions" insertion point to make the v0.3.1 addition non-breaking.
- **Guardian designation surface entirely.** The new recovery model is Codex Identity seed-derivation. If a future Mnemosyne version wants Guardians as an extra recovery layer, they ship in a separate version (TBD).
- **StoicTag local-state.** Chain is the source of truth; consumers read on demand via free `resolve-stoictag` chain reads. A v0.3.x optimization may add a tiny client-side cache helper if reads become a bottleneck.
- **UiSettings final cleanup** (removing OuronetUI-specific fields from generic `UiSettings`). Ships as v0.3.1 housekeeping after Mnemosyne v0.1 is live. The v0.3.0 migration already relocates these into `consumerSettings["OuronetUI"]` so the v0.3.1 cleanup is a safe field removal.
- **Movie-platform / Demiourgos Streaming Platform consumer slot.** Added when that consumer ships its codex integration; the registry is ready to accept new consumer names without a package bump.
