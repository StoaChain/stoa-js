# @stoachain/ouronet-codex v0.3.0 — Design Lock

**Status:** discussion-phase design (2026-05-27); not yet implemented
**Author:** Claude (Epyc session, 2026-05-27), based on Mnemosyne design discussion
**Drives:** unblocks Mnemosyne v0.1 — see `Mnemosyne/docs/v0.1-design.md` §9 for the full Mnemosyne context

This document specifies the v0.3.0 changes to `@stoachain/ouronet-codex`. All changes are **additive** — every v0.2.0 export keeps its signature. No breaking changes.

---

## 1. Why v0.3.0 happens

v0.3.0 ships the codex-package features that **Mnemosyne v0.1 requires** to function as designed. Specifically:

1. **Guardian designation** — Mnemosyne's recovery flow relies on Schnorr-verify-against-on-chain-pubkey for the CodexPrime AND for designated secondary accounts. The package needs a way to mark accounts as Guardians.
2. **Activation status tracking** — Mnemosyne (and other consumers) need to know whether a Guardian has been activated on-chain (a prerequisite for the Schnorr-recovery primitive to work). The package needs to cache this status.
3. **Consumer settings registry** — With multiple consumers (OuronetUI, AH hub, Mnemosyne, future apps) sharing a single codex, the loosely-typed `UiSettings[extra: string]: unknown` escape hatch becomes a collision risk. The package needs proper namespacing.
4. **Schema migration framework** — As the codex schema evolves across consumers and versions, we need a formal migration mechanism instead of ad-hoc per-consumer drift handling.

Items 1-3 are Mnemosyne-blocking. Item 4 is hygiene that becomes essential once multiple consumers write to the same codex.

A 5th item — **UiSettings cleanup** (removing OuronetUI-specific fields from the generic `UiSettings` interface) — is housekeeping that can ship as v0.3.1 after v0.3.0 lands, since Mnemosyne doesn't strictly require it.

## 2. Type changes (additive)

### 2.1 `IOuroAccount` — add `isGuardian`, `isActivated`, `activationCheckedAt`

```typescript
export interface IOuroAccount {
  // ... existing v0.2.0 fields ...
  isPrime?: boolean;
  parentSeedId?: string;

  /** Guardian designation (v0.3.0+). True if this account has been
   *  designated as a recovery Guardian. Alongside the CodexPrime, this
   *  account can prove codex ownership via Schnorr signature verification
   *  against its on-chain public key — useful for Mnemosyne-style
   *  cloud-stored codices where the user may forget their password but
   *  remember a Guardian's seed.
   *
   *  Constraints (enforced by designateGuardian action):
   *    - account MUST exist in the codex
   *    - account.isSmart MUST === false (Standard Ouronet Accounts only)
   *    - account MUST NOT be the CodexPrime (CodexPrime is implicitly a
   *      Guardian; the explicit flag is for ADDITIONAL Guardians)
   *
   *  Activation requirement: Guardian-based recovery only works against
   *  ACTIVATED accounts (those whose pubkey is on-chain). The designation
   *  itself is allowed pre-activation, but consumers should warn users
   *  if they designate a not-yet-activated account.
   *
   *  No M-of-N at v0.3.0 — any single Guardian's signature is sufficient
   *  for recovery. M-of-N could come in a later release. */
  isGuardian?: boolean;

  /** Cached activation status (v0.3.0+). When the consumer last checked
   *  the on-chain state, was this account activated (i.e., does its
   *  public key appear in a Stoa account record)?
   *
   *  Semantics:
   *    undefined / null — never checked, status unknown
   *    true — activated as of `activationCheckedAt`
   *    false — checked and confirmed not activated as of `activationCheckedAt`
   *
   *  The package does NOT make chain calls itself (it's chain-agnostic
   *  at this layer). Consumers (OuronetUI, Mnemosyne SPA, etc.) check
   *  on-chain and update the cache via the activation-status hook. */
  isActivated?: boolean | null;

  /** ISO timestamp of last activation-status check (v0.3.0+). Consumers
   *  can use this to decide whether to refresh (e.g., refresh if older
   *  than 1 hour). */
  activationCheckedAt?: string;
}
```

### 2.2 `IConsumerSettings` — new type

```typescript
/** Per-consumer settings entry within the codex's consumerSettings
 *  registry (v0.3.0+). Each consumer (OuronetUI, AncientHoldings,
 *  Mnemosyne, MoviePlatform, etc.) owns ONE entry keyed by its
 *  canonical name and writes ONLY to its own entry. Other entries
 *  are preserved verbatim on save.
 *
 *  Replaces the v0.1.0 / v0.2.0 pattern of dumping consumer-specific
 *  keys into `UiSettings[extra: string]` — that pattern produces
 *  collisions when multiple consumers share a codex via Mnemosyne. */
export interface IConsumerSettings {
  /** Canonical consumer identifier (e.g. "OuronetUI", "AncientHoldings",
   *  "Mnemosyne", "MoviePlatform"). MUST be unique across consumers
   *  reading the same codex. By convention: PascalCase, no spaces. */
  consumerName: string;

  /** Semver version of the consumer that last wrote these settings.
   *  Used by future readers of this consumer's settings to detect
   *  outdated writes (e.g., if a user opened the codex from an old
   *  OuronetUI version after recently using a new one). */
  consumerVersion: string;

  /** Schema version internal to THIS consumer's settings shape.
   *  Each consumer evolves its settings independently; this is NOT
   *  the codex's overall schemaVersion. Bumped when the consumer's
   *  settings interface changes shape. */
  schemaVersion: number;

  /** Opaque-to-other-consumers settings blob, typed by the consumer.
   *  Other consumers MUST preserve this verbatim on save (additive
   *  convention — never drop fields you don't recognize). */
  settings: Record<string, unknown>;

  /** ISO timestamp of last write. */
  lastUpdatedAt: string;
}
```

### 2.4.bis `IPrimePureKey` — new (2026-05-28 Mnemosyne-Ledger refinements pass)

Auto-generated Kadena ED25519 keypair stored as an immutable, undeletable part of every Mnemosyne-bound codex. Serves as the **on-chain signing authority** for codex management operations when codex contents eventually move on-chain (v1.0 of Mnemosyne). Users never input this key; the system generates it at codex creation, shows the public key once, and stores the private key encrypted inside the codex.

```typescript
/** The codex's auto-generated signing keypair (v0.3.0+, Mnemosyne-bound
 *  codices only). Used as the on-chain authorization guard for codex-
 *  management operations.
 *
 *  Generated at codex creation; IMMUTABLE thereafter. No update or delete
 *  action exists for this entity — it's structurally part of the codex's
 *  identity layer alongside the user-controlled Codex Identity (Apollo).
 *
 *  The public key is registered on-chain as the `codex-guard` field of
 *  the Codex Identity entry in the mnemosyne-ledger Pact module. See
 *  Mnemosyne v0.1 design doc for the on-chain registration semantics
 *  and the rationale for the Apollo+PrimePureKey dual-key architecture.
 *
 *  Why a separate Kadena key (not just the Apollo identity)?
 *  Pact's enforce-keyset / enforce-guard operate on Kadena ED25519 keys.
 *  Apollo signatures cannot satisfy Kadena keyset guards (different
 *  curve, no native Pact verifier). The PrimePureKey bridges the two
 *  worlds — Apollo is for user-recognizable identification, PrimePureKey
 *  is for Pact-native authorization. */
export interface IPrimePureKey {
  /** 32-byte Kadena ED25519 public key (hex-encoded). Stored on-chain in
   *  the codex-guard field of the Codex Identity entry. */
  publicKey: string;

  /** The Kadena ED25519 private key, encrypted with the codex's Codex
   *  Key (CK) using AES-256-GCM. Decryptable only by anyone who can
   *  unlock the codex (i.e., knows password / Apollo identity privkey /
   *  Prime Codex Seed mnemonic). */
  encryptedPrivateKey: string;

  /** ISO timestamp of when this PrimePureKey was generated. */
  generatedAt: string;
}

// On CodexSnapshot:
export interface CodexSnapshot {
  // ... existing fields ...

  /** Mandatory for codices created via Mnemosyne-aware kickstartCodex
   *  (v0.3.0+ with `generatePrimePureKey: true` option). Optional for
   *  legacy codices migrated from pre-Mnemosyne OuronetUI — they receive
   *  one auto-generated during the migration flow. Once present, IMMUTABLE.
   *
   *  NEVER expose this field to users; it's a system-managed key. The
   *  codex package surfaces ONLY the public key for the user-visible
   *  "Your codex signing key is X" copy at signup. */
  primePureKey?: IPrimePureKey;
}
```

**Constraint summary (enforced by store actions, see §3.4 below):**

- Exactly ONE `primePureKey` per codex (or none, for legacy)
- Generated atomically during `kickstartCodex({ ..., generatePrimePureKey: true })`
- NEVER replaceable, deletable, or updatable — there is no `replacePrimePureKey` / `deletePrimePureKey` action by design
- The encrypted private key is wrapped with the same CK that wraps the rest of the codex; unlocking the codex transparently unlocks PrimePureKey
- The public key is what gets sent to the on-chain `mnemosyne-ledger.register-codex-identity` call as the `codex-guard` field

### 2.4 `IHerotag` — new (2026-05-27 refinements pass)

A new codex-level field for the user's chosen herotag — a globally-unique human-readable handle (e.g. `@mihai`) that resolves to a specific ouro address. **The herotag registry itself lives on Stoa-chain** in an immutable Pact module (out-of-scope for this package; lives in Mnemosyne / cross-stoa-ecosystem deployment). This codex field is a cache + local metadata for the on-chain registration.

```typescript
/** The user's chosen herotag (v0.3.0+). At most one per codex.
 *  Globally unique in the on-chain registry. Immutable target —
 *  the on-chain Pact module enforces this (no update function;
 *  only insert). See Mnemosyne v0.1 design doc §5.3 for the on-chain
 *  module specification and §9.1.3 for the in-codex semantics. */
export interface IHerotag {
  /** Without the @ prefix. Lowercased, alphanumeric + dots + hyphens,
   *  3-30 chars. Globally unique across all Mnemosyne users
   *  (uniqueness enforced cryptographically by Pact's `insert`). */
  name: string;

  /** The ouro address this herotag resolves to. MUST be a Standard
   *  Ouronet Account (isSmart: false) AND activated on-chain at the
   *  time the herotag was claimed. Defaults to CodexPrime if user
   *  didn't specify another at claim time. IMMUTABLE post-claim. */
  targetOuroAddress: string;

  /** ISO timestamp of when the herotag was claimed. */
  claimedAt: string;

  /** The Stoa-chain tx hash that registered the claim. Cached so
   *  consumers can verify the on-chain record without re-querying. */
  onChainTxId?: string;

  /** True if the package has confirmed the herotag is present in
   *  the on-chain registry. False if pending confirmation. Updated
   *  by consumer-triggered chain verification calls. */
  onChainConfirmed: boolean;

  /** ISO timestamp of last on-chain verification. */
  lastChainCheck: string;
}

// On CodexSnapshot:
export interface CodexSnapshot {
  // ... existing fields ...
  herotag?: IHerotag;  // singular — at most one per codex
}
```

**Constraint summary** (enforced by `recordHerotagClaim` action, see §3.3):
- One herotag per codex maximum (`already-claimed` error if codex already has one)
- Target account must exist in the codex (`target-not-in-codex` error)
- Target account must be `isSmart === false` (`target-not-standard` error)
- Target account must be activated on-chain (`target-not-activated` error — checked via the `isActivated` cache from §2.5)
- Name must match format `[a-z0-9.-]{3,30}` (`invalid-format` error)

### 2.5 No changes to `IKadenaSeed`, `IPureKeypair`, `AddressBookEntry`, `WatchListEntry`, `UiSettings`

v0.3.0 leaves these untouched. The v0.3.1 (or later) UiSettings cleanup removes OuronetUI-specific fields from `UiSettings` and migrates them to `consumerSettings["OuronetUI"]`, but that's separate.

### 3.1 `CodexGuardianError`

```typescript
/** Thrown by Guardian-related operations when validation fails. */
export class CodexGuardianError extends CodexError {
  public override readonly name = "CodexGuardianError";
  public readonly accountId: string;
  public readonly reason:
    | "smart-account-not-allowed"
    | "account-not-found"
    | "already-prime"
    | "not-guardian";  // un-designate called on a non-Guardian

  constructor(reason: CodexGuardianError["reason"], accountId: string, detail?: string) {
    const messages: Record<CodexGuardianError["reason"], string> = {
      "smart-account-not-allowed":
        `Account ${accountId} is a Smart Ouronet Account; Guardian designation requires Standard accounts only.`,
      "account-not-found":
        `Account ${accountId} does not exist in this codex.`,
      "already-prime":
        `Account ${accountId} is the CodexPrime; it is implicitly a Guardian and does not need explicit designation.`,
      "not-guardian":
        `Account ${accountId} is not currently designated as a Guardian.`,
    };
    super(detail ? `${messages[reason]} ${detail}` : messages[reason]);
    this.reason = reason;
    this.accountId = accountId;
  }
}
```

### Updated `kickstartCodex` signature (v0.3.0+)

```typescript
interface KickstartArgs {
  /** Pre-formed kadena seed. (v0.2.0+) */
  seed: Omit<IKadenaSeed, "isPrime">;
  /** Pre-formed CodexPrime ouro account. (v0.2.0+) */
  primeOuroAccount: Omit<IOuroAccount, "isPrime" | "parentSeedId">;
  /** Generate a PrimePureKey alongside the prime entities (v0.3.0+).
   *  Set to true for Mnemosyne-bound codices; false (or omit) for codices
   *  that don't intend to register on the Mnemosyne Ledger.
   *  Default: false (backward-compatible with v0.2.0 callers). */
  generatePrimePureKey?: boolean;
}

interface KickstartResult {
  seed: IKadenaSeed;             // with isPrime: true
  primeOuro: IOuroAccount;       // with isPrime: true + parentSeedId
  primePureKey?: IPrimePureKey;  // populated iff generatePrimePureKey was true
}
```

When `generatePrimePureKey: true`:
1. `kickstartCodex` generates a fresh ED25519 Kadena keypair (via the codex package's `@stoachain/kadena-stoic-legacy/cryptography-utils` import, or a similar primitive)
2. The public key becomes the `primePureKey.publicKey` (visible to consumer)
3. The private key is encrypted with the codex's CK and stored as `primePureKey.encryptedPrivateKey`
4. The full `IPrimePureKey` is added to the snapshot atomically (same transaction as the other prime entities)
5. The consumer (e.g., Mnemosyne SPA) then takes the public key and uses it as the `codex-guard` field when calling `mnemosyne-ledger.register-codex-identity` on Stoa chain

### `deleteOuroAccount` / `deleteKadenaSeed` extensions

Same v0.2.0 behavior — they only reject deletion of CodexPrime / Prime Codex Seed.

### NEW immutability surface: no `deletePrimePureKey` action exists

By design. The PrimePureKey is structurally part of the codex's identity layer; removing it would orphan the on-chain Codex Identity entry's `codex-guard` field. The only way to "remove" a PrimePureKey is to delete the entire codex (codex-reset flow). Consumers MUST NOT attempt to bypass this; the codex package exposes no API for it.

### 3.3.bis Arweave backup + Mnemosyne-Codex chain pointer (2026-05-28 Mnemosyne-Ledger pass)

The codex package owns the codex-management lifecycle, including the off-chain backup flow that lets a user pin their encrypted codex to Arweave and update the `mnemosyne.CODEX` Pact module's pointer. Keeping these actions inside `ouronet-codex` (rather than splitting them into a separate `mnemosyne-client` package) reflects the owner's framing: **Arweave backup is a codex-management feature**, not a separate Mnemosyne-specific concern. Consumers that don't use Mnemosyne can ignore the actions; they're side-effect-free if never called.

New cached entity (lives in the codex itself, not on the snapshot top-level — it's metadata about the codex's own backups):

```typescript
/** Local cache of past Arweave backups, for offline display in the SPA
 *  without hitting the chain on every render. Synced from the
 *  mnemosyne.CODEX.codex-arweave-history table on demand via
 *  `refreshArweaveBackupHistory()`. Append-only; never trimmed. */
export interface IArweaveBackupEntry {
  /** Arweave transaction id (43-char base64url). */
  arweaveTxId: string;
  /** ISO timestamp when the backup was confirmed on Stoa chain. */
  uploadedAt: string;
  /** Optional size-in-bytes of the uploaded blob, for SPA display.
   *  Populated locally at upload time; not stored on-chain. */
  uploadedSizeBytes?: number;
}

// On CodexSnapshot:
export interface CodexSnapshot {
  // ... existing fields ...
  arweaveBackupHistory?: IArweaveBackupEntry[];  // append-only local cache
}
```

New error class:

   *  for confirming inclusion before calling this. */

**Rationale for keeping these in `ouronet-codex` (vs a separate `mnemosyne-client` package):**

The owner's framing: Arweave backup is a codex-management feature, sitting alongside Guardian designation, herotag claiming, and PrimePureKey lifecycle. All four are "things that affect how your codex is stored or recovered." Splitting Arweave out into a separate package would force consumers who want Mnemosyne integration to add a second dependency for what's really one feature surface.

Trade-off accepted: consumers who DON'T use Mnemosyne carry the Arweave-related types and actions but pay no runtime cost (the actions are pure functions that never auto-execute). The Bundlr/Irys SDK itself is NOT a dependency of the codex package — the consumer-supplied `uploader` callback in `useArweaveBackup.backupNow()` is where Bundlr lives. The codex package only knows about encrypted blobs and Pact tx construction; it never imports an Arweave SDK.

**Rationale for splitting upload from chain-update:**

The Mnemosyne SPA needs to display a payment-confirmation UI BETWEEN getting the ciphertext from the codex package AND uploading to Arweave (the user might cancel after seeing the AR cost). Likewise, between Arweave upload completing AND submitting the chain tx, the SPA wants to show "uploaded, now finalizing on chain" feedback. By exposing `exportEncryptedSnapshotForArweave` + `buildUpdateArweavePointerTx` as separate actions, the SPA can choreograph the UX without the codex package needing to know about Bundlr/Irys or React UI patterns.

### 3.4 Schema migration framework

```typescript
// In a new file: src/state/migrations.ts

export interface SchemaMigration {
  fromVersion: number;
  toVersion: number;
  migrate: (snapshot: CodexSnapshot) => CodexSnapshot;
  description: string;
}

/** Registry of all schema migrations. Applied in order on init() when
 *  the loaded snapshot's schemaVersion is older than the package's
 *  current version. Migrations are pure functions — no I/O. */
export const SCHEMA_MIGRATIONS: SchemaMigration[] = [
  // v0.3.0 introduces consumerSettings field; pre-v0.3.0 snapshots
  // have it undefined. The migration is a no-op write of an empty
  // object to formalize the field's existence.
  {
    fromVersion: 0,
    toVersion: 1,
    migrate: (snap) => ({ ...snap, consumerSettings: snap.consumerSettings ?? {} }),
    description: "v0.3.0: initialize consumerSettings registry",
  },
  // Future migrations append here. Never delete past entries.
];

/** Current schema version this package writes. Bumped per migration. */
export const CURRENT_SCHEMA_VERSION = 1;

/** Returns true if the consumer's package version can safely write a
 *  codex at the given loaded schema version. Returns false if the loaded
 *  codex is at a NEWER schema (consumer would risk dropping unknown
 *  fields). */
export function canConsumerWrite(loadedSchemaVersion: number): boolean {
  return loadedSchemaVersion <= CURRENT_SCHEMA_VERSION;
}
```

## 4. Behavior changes (no breaking changes)

### 4.1 `actions.init(adapter)` — extend with migration

The existing init flow (which already does the v0.2.0 legacy-codex migration for Prime Codex Seed flagging) gets a second migration step at the same point:

```typescript
async init(adapter, deviceVariant) {
  // ... existing v0.2.0 init logic ...

  const snap = await adapter.loadAll();

  // v0.2.0 legacy migration (Prime Codex Seed) — unchanged
  // ...

  // v0.3.0 NEW: apply schema migrations from SCHEMA_MIGRATIONS
  let migrated = migratedKadenaSeeds ? { ...snap, kadenaSeeds: migratedKadenaSeeds } : snap;
  const fromVersion = snap.schemaVersion ?? 0;
  if (fromVersion < CURRENT_SCHEMA_VERSION) {
    for (const m of SCHEMA_MIGRATIONS) {
      if (m.fromVersion >= fromVersion && m.toVersion <= CURRENT_SCHEMA_VERSION) {
        migrated = m.migrate(migrated);
      }
    }
    migrated = { ...migrated, schemaVersion: CURRENT_SCHEMA_VERSION };
    // Persist eagerly so the upgrade sticks
    try {
      await adapter.saveAll(migrated);
    } catch {
      // Best-effort
    }
  }

  set({ /* ... use migrated instead of snap ... */ });
}
```

### 4.2 `actions.deleteOuroAccount(id)` — extend Guardian handling

The existing CodexPrime protection (`isPrime: true` throws `CodexPrimeProtectedError`) stays. Add: deleting a Guardian-designated account is ALLOWED (un-designation happens automatically as a side effect). Optionally: emit a warning console log for transparency.

### 4.3 `addOuroAccount` — preserve existing flags + new fields

When upserting an account that already exists (by id), preserve `isGuardian` if not explicitly set in the new account. Same pattern as existing `parentSeedId` validation.

## 6. Backward compatibility matrix

| Reader \ Writer | v0.1.0 | v0.2.0 | v0.3.0 |
|---|---|---|---|
| v0.1.0 reads | ✓ | ✓ (ignores new fields) | ✓ (ignores new fields) |
| v0.2.0 reads | ✓ | ✓ | ✓ (ignores new fields) |
| v0.3.0 reads | ✓ (migration applied) | ✓ (migration applied) | ✓ |
| v0.1.0 writes | (consumed by all) | (consumed by all) | (consumed by all; v0.3.0 migrates on read) |
| v0.2.0 writes | (preserves v0.3.0 fields via additive convention) | (canonical) | (consumed; v0.3.0 migrates on read) |
| v0.3.0 writes | ⚠️ v0.1.0 reader doesn't understand consumerSettings/Guardians | same | (canonical) |

**Key invariant:** unknown fields are preserved verbatim on read+save by all versions (additive convention). So a v0.1.0 client that reads a v0.3.0 codex, makes a small change, and writes back, **does not drop** the Guardian flags or consumerSettings — they round-trip even through old readers.

`canConsumerWrite(loadedSchemaVersion)` lets a writer EXPLICITLY refuse to write if it doesn't understand the schema. This is opt-in for paranoid consumers; default is the additive-preservation path.

## 8. Out of scope (deferred to later)

- **UiSettings cleanup** (remove OuronetUI-specific fields from generic `UiSettings`). Ship as v0.3.1 after v0.3.0 lands; not Mnemosyne-blocking.
- **M-of-N Guardian recovery** (require multiple Guardians to co-sign for recovery, à la Argent's social recovery threshold). Not needed at v0.3.0; consider for v0.4.0+ if Mnemosyne adds advanced recovery features.
- **Activation triggering** (the package doesn't try to ACTIVATE accounts on-chain — that's a chain-side operation that consumers handle, e.g. Mnemosyne triggers via Ouronet Gas Station, OuronetUI triggers via the CFM modal flow).
- **Built-in Mnemosyne client integration** (no `MnemosyneCloudAdapter` in the codex package itself — that ships as a separate `@stoachain/mnemosyne-client` package so consumers not using Mnemosyne don't pull the dep).
- **Codex package CodexProvider props for Mnemosyne** (e.g., recovery-flow UI components). If needed, add in v0.3.x; Mnemosyne can ship its own UI initially.
- **The on-chain herotag Pact module itself.** This package only carries the `IHerotag` field + actions for caching local state about the on-chain registration. Writing/deploying the actual `mnemosyne-herotag` Pact module lives in the Mnemosyne project (or a separate Stoa-ecosystem deployment), NOT in the codex package. The codex package is React/TypeScript; Pact module deployment is operational. Separate concern.
- **Herotag-claim chain tx submission.** The codex package's `recordHerotagClaim` action records the LOCAL state after the consumer has already submitted the chain tx. The consumer (Mnemosyne SPA) owns the chain-side flow — building the Pact tx, signing with the target account's keys, submitting to a Stoa node, polling for confirmation. The package stays chain-client-agnostic.
